# تقدم مشروع ELITE WORLD

آخر تحديث: 1 يوليو 2026

## ما تم تنفيذه

- تأسيس Next.js 16.2.9 وReact 19.2.7 وTypeScript Strict وTailwind CSS 4.3.
- نقل الشعار الأصلي واستخدامه في الهيدر والفوتر وشاشة التحميل وFavicon.
- تحليل ألوان الشعار الفعلية وتأكيد `#0099BF` و`#00677F` و`#FFFFFF`.
- إنشاء نظام تصميم مركزي وخطوط محلية وRTL كامل.
- إنشاء الهيدر الثابت، الشريط العلوي، قائمة جوال قابلة للإغلاق بلوحة المفاتيح، وفوتر تفاعلي.
- إنشاء شريط تنقل سفلي للجوال.
- تنفيذ الصفحة الرئيسية بكل الأقسام المطلوبة وبالترتيب.
- إنشاء 10 أقسام، 8 منتجات، 3 مشاريع، 3 تقييمات، 3 مقالات، و6 علامات تجريبية من ملفات بيانات منفصلة.
- إنشاء Toast للمفضلة والمقارنة، ونموذج نشرة محلي مع Validation ورسالة نجاح.
- إنشاء 15 صفحة Placeholder مصممة، إضافة إلى صفحة المقارنة وصفحة 404.
- إضافة Metadata وOpen Graph وCanonical وOrganization Schema وSitemap وRobots.
- إضافة Skip Link وSemantic HTML وARIA وFocus States واحترام Reduced Motion.
- إضافة صورة Hero محلية ورسوم هندسية SVG محلية دون روابط صور خارجية.
- إضافة توثيق التشغيل والتخصيص ونظام التصميم.

### المرحلة الثانية — الكتالوج المحلي

- إنشاء 8 أقسام رئيسية مع أقسام فرعية منظمة.
- إنشاء 20 منتجاً تجريبياً بأسماء عربية وإنجليزية وSKU وموديلات ومواصفات فنية.
- تحويل `/shop` إلى متجر فعلي ببحث وفلاتر وترتيب وQuery Parameters.
- تحويل `/categories` وإضافة `/categories/[slug]` لعرض الأقسام والمنتجات.
- إضافة `/products/[slug]` بتفاصيل كاملة وجدول مواصفات ومنتجات مشابهة.
- تحويل `/search` إلى بحث محلي يدعم العربية والإنجليزية وSKU والموديل والعلامة والقسم.
- تطوير `ProductCard` الحالية وربطها بصفحات المنتجات وواتساب.
- إضافة معالجة واتساب تمنع الرابط الخاطئ عند بقاء الرقم Placeholder.
- توحيد بيانات الصفحة الرئيسية والمتجر والأقسام حول مصدر الكتالوج نفسه.

### المرحلة الثالثة — قاعدة بيانات الكتالوج

- إضافة Prisma 7 مع PostgreSQL ومخطط منظم لأربعة نماذج: `Category` و`Product` و`ProductSpecification` و`Brand`.
- إضافة Migration أولي قابل للتطبيق على PostgreSQL.
- إضافة Seed قابل لإعادة التشغيل ينقل بيانات Demo الحالية: 8 أقسام و20 منتجاً مع العلامات والمواصفات، دون تغيير Slug أو SKU أو المسارات.
- إنشاء `catalog repository/service` وعزل صفحات الموقع عن القراءة المباشرة لملفات البيانات.
- ربط المتجر والبحث وصفحات الأقسام والمنتجات والصفحة الرئيسية وSitemap بطبقة البيانات الجديدة دون تغيير التصميم.
- كان المشروع يدعم fallback محلياً في هذه المرحلة؛ أُلغي لاحقاً عند توحيد مصدر التشغيل على Prisma.
- إضافة أوامر توليد Prisma Client وتطبيق Migration وتشغيل Seed وفتح Prisma Studio.

- إضافة تأكيد قبل الحذف ورسائل نجاح وفشل واضحة.

- إضافة `/admin/categories` لإدارة الاسم العربي والإنجليزي وSlug والوصف والصورة والترتيب.
- إضافة `/admin/brands` لإدارة الاسم وبلد المنشأ والوصف الاختياريين.

- منع حذف أي قسم أو علامة مرتبطة بمنتجات، مع رسالة واضحة في الواجهة وفحص داخل عملية الحذف.
- ربط نموذج المنتج بقائمتي الأقسام والعلامات من Prisma مباشرة دون إنشاء علامات ضمن نموذج المنتج.
- إضافة إعادة تحقق تلقائية للرئيسية والمتجر والبحث وصفحات الأقسام والمنتجات وSitemap بعد عمليات الإدارة.

- الإبقاء على CSV والصور والمنتجات وSeed الحالية دون حذف أو إعادة تسمية، وعدم تشغيل Seed أثناء التنفيذ.

### المرحلة السادسة — إدارة صور المنتجات

- إضافة نموذج `ProductImage` وMigration تراكمي يسجل الصور الرئيسية الحالية دون حذفها أو تغيير مساراتها.
- إضافة Storage Adapter لمعالجة JPG وPNG وWebP، مع Vercel Blob لصور الإنتاج وتخزين محلي أثناء التطوير دون تغيير لوحة التحكم.
- ضغط الصور وتحويلها إلى WebP بجودة 86% وحد أقصى 2400×2400 بكسل، مع حد 8 MB للصورة و8 صور للعملية.
- إضافة معاينة قبل الرفع وAlt Text عربي إلزامي وتعيين الصورة الرئيسية وتغيير ترتيب الصور وحذف الارتباط بتأكيد.
- حماية ملفات الصور الأصلية الحالية من الحذف الفعلي؛ تنظيف الملفات محصور في مجلد `uploads` الجديد وبعد التأكد من عدم استخدامها.
- إضافة `/admin/media` لعرض الصور المستخدمة والبحث باسم المنتج أو SKU أو Alt Text أو المسار.
- ربط صور Prisma ببطاقات المنتجات ومعرض صفحة المنتج، مع إعادة تحقق تلقائية لتظهر التغييرات فوراً.
- تحديث Seed بصورة تراكمية: يضيف ميتاداتا الصور فقط عند عدم وجودها ولا يمس صور الإدارة، ولم يتم تشغيل Seed أثناء التنفيذ.

- إضافة `/admin/seo` لتدقيق الحقول التلقائية والعناوين والأوصاف المكررة وإدارة الصفحات العامة دون توليد جماعي.

### المرحلة العاشرة — السلة وCheckout

- إضافة سلة مشتريات فعلية محفوظة محلياً في المتصفح دون حسابات مستخدمين.
- إضافة `/cart` لتعديل الكميات والحذف وعرض الإجمالي المبدئي.
- إضافة `/checkout` بخيارين: طلب عرض سعر أو طلب مباشر.
- الطلب المباشر يُنشأ فقط للمنتجات ذات الأسعار المحددة، ويُمنع عند وجود سعر ناقص.
- دعم الطلب المباشر باستخدام نماذج المرحلة التاسعة نفسها داخل معاملة واحدة، مع سجل داخلي مخفي عن قائمة عروض الأسعار ودون Migration جديدة.
- إضافة زر «اطلب الآن» وأزرار السلة وعداد فعلي في الهيدر والتنقل السفلي.
- عرض مصدر الطلب في لوحة الإدارة، مع استمرار التتبع والفاتورة والمدفوعات الداخلية كما هي.
- لا توجد بوابة دفع أو شحن API أو حسابات مستخدمين، ولم يتم تشغيل Seed أو Migration على قاعدة فعلية. Stage 10 لا يحتاج Migration إضافية بعد تطبيق Migrations المرحلة التاسعة.

### المرحلة الحادية عشرة — نظام دفع Modular

- إضافة نموذج Payment قابل للتوسع: provider وproviderRef وmethod وmetadata وcurrency وحالات `PENDING/PAID/FAILED/CANCELLED`.
- إضافة Payment Provider Interface وMock Provider وProvider Factory دون أي SDK أو API خارجي.
- إضافة Payment Service لا يعتمد على مزود محدد، مع `createPayment` و`processPayment` و`updateStatus`.
- ربط الطلب المباشر في Checkout بإنشاء Payment تلقائي عبر المزود الافتراضي `CHECKOUT_PAYMENT_PROVIDER=mock`.
- إضافة `/api/payments/create` و`/api/payments/verify`.
- عرض Provider وTransaction Reference وStatus وAmount داخل تفاصيل الطلب في الإدارة.
- لا توجد Stripe أو Moyasar أو بوابة دفع حقيقية، ولم يتم تشغيل Seed أو Migration على قاعدة فعلية.

### المرحلة الثانية عشرة — مزودات الدفع والـ Webhooks

- إزالة الاعتماد على `generated/prisma` من الكود، والاعتماد على `@prisma/client` فقط.
- تحديث Payment إلى علاقة طلب واحد مع سجل دفع واحد واضح، وإضافة `webhookEventId` و`webhookReceivedAt`.
- إضافة هيكل Stripe Provider وMoyasar Provider دون SDK ودون أي طلب خارجي.
- تحديث Provider Factory لدعم `mock` و`stripe` و`moyasar`.
- جعل PaymentService المصدر الوحيد لتحديث حالة الدفع ومزامنة الطلب.
- إضافة Webhook endpoint موحد: `/api/payments/webhook` مع دعم provider عبر query/body.
- جعل معالجة الـ Webhook idempotent عبر حفظ `webhookEventId` وعدم تكرار نفس الحدث.
- مزامنة الحالات: `PENDING → PENDING_PAYMENT`، و`PAID → CONFIRMED`، و`FAILED/CANCELLED → CANCELLED`.
- عرض مزود الدفع وحالة الدفع والمرجع والمبلغ وحالة Webhook في قائمة وتفاصيل الطلبات.
- لم يتم تفعيل Stripe أو Moyasar فعلياً، ولم يتم تشغيل Seed أو Migration على قاعدة فعلية.

## القرارات التقنية

### المرحلة الثامنة والعشرون — Product Import / Export Center

- المصدر التشغيلي للمنتجات يبقى Prisma، وExcel أداة نقل وإدارة جماعية لا بديلًا لقاعدة البيانات.
- التصدير يحافظ على Product ID وSKU/slug ومراجع الصور والمواصفات وحقول SEO، ويضيف صورة مصغرة عند إمكان قراءتها.
- التحليل يحفظ Preview فقط، ولا يبدأ أي كتابة قبل التأكيد. التنفيذ بدفعات من 50 داخل Transactions مستقلة مع Snapshot للصفوف المطبقة.
- التراجع يستخدم مقارنة `updatedAt` ذرية، ولا يحذف منتجًا أنشأته العملية إذا ارتبط لاحقًا بطلب أو عرض أو سلة أو مفضلة أو نشاط.
- الروابط الخارجية للصور محصورة في HTTPS العام مع منع العناوين المحلية وحد للحجم والنوع، وZIP لا يُستخرج إلى نظام الملفات.
- الحد الحالي للتحليل 10,000 صف و20MB لـXLSX و50MB لـZIP. للمكتبات الأكبر أو المهام التي تتجاوز مهلة Vercel يلزم لاحقًا عامل خلفي/Queue مع إبقاء صيغة السجل الحالية قابلة للترقية.
- Migration سجل العمليات إضافية فقط، وقد طُبقت على Production بعد مراجعتها دون حذف أو تعديل أي بيانات حالية.

### المرحلة السابعة والعشرون — التسوق حسب النشاط

- نموذج كامل لنشاط «مطعم برجر» مع Landing Page بيعية وSEO وStructured Data وربط داخلي ومجموعات معدات واضحة.
- علاقة many-to-many بين الأنشطة والمنتجات تسمح للمنتج الواحد بالظهور في عدة أنشطة، مع إدارة النشر والمحتوى والربط من لوحة التحكم.
- إعادة استخدام ProductCard والكتالوج وطلب عرض السعر وواتساب دون تكرار بيانات المنتجات، وإضافة المنتجات المكملة إلى صفحة المنتج من روابط الأنشطة.
- لم تُعمم بقية الأنشطة بعد؛ تُضاف من لوحة الإدارة بعد اعتماد نموذج البرجر.

- App Router مع Server Components افتراضياً لتقليل JavaScript.
- Client Components محصورة في التفاعلات التي تحتاج المتصفح مثل السلة والفلاتر والقوائم.
- بيانات الشركة الافتراضية في `config/site.ts`، بينما الإعدادات المعروضة تأتي من `SiteSettings` عند توفر Prisma.
- ملف CSV أداة استيراد وSeed فقط، بينما Prisma هو مصدر بيانات الكتالوج وقت التشغيل.
